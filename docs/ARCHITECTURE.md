# Starline Architecture

## 目标

Starline 首版聚焦教培学习闭环：

1. 后台创建学习套餐。
2. 给学生开通套餐。
3. 系统自动开放课程、课件资料和小挑战。
4. 学生在小程序学习、提交小挑战。
5. 老师在后台批改并发送反馈。

## 分层

```text
learning-api/
  cmd/api
  internal/interfaces
  internal/application
  internal/domain
  internal/infrastructure

web/
  React 管理后台

miniprogram/
  微信小程序学生端
```

## 关键约定

- API 前缀统一为 `/api`。
- 用户统一通过微信身份登录，后端用自有用户、角色和数据范围做最终鉴权。
- 管理端操作人从登录 token 解析，写操作继续落到操作日志。
- 套餐负责售卖和权限，资料归属于学年、年级、学期、学科、课程、章节。
- 开通套餐前必须返回“本次开放内容预览”。

## 基础权限

- 学生只能访问自己的 `/api/student/*` 数据，学习内容由已开通套餐决定。
- 教师默认按课程/班级范围查看学生、作业、批改和通知。
- 后台固定三档管理角色：运营教务、校区管理员、超级管理员。
- 前端隐藏无权限菜单，后端中间件仍会对接口做登录和角色校验。

## 学习数据权限

学习内容权限参考 Circle 的“成员 - 空间 - 访问组”模型落地：

- 学生对应成员，`students` 是被授权对象。
- 学习空间对应空间，`learning_spaces` 固定为“学年 + 年级 + 学科 + 学期 + 阶段”。
- 套餐对应访问组，`study_packages` 是售卖和授权入口。
- `package_spaces` 定义套餐开放哪些学习空间。
- `package_content_types` 定义套餐开放哪些内容类型：题、讲义、课程。
- `student_package_grants` 记录学生开通了哪个套餐。
- `student_learning_space_access` 是派生权限表，记录学生实际拥有的学习空间访问权。

学生端查询内容时不直接按套餐学科硬编码判断，而是按“学生学习空间访问 + 套餐内容类型 + 内容所属学习空间”过滤。后台数据权限页保留学生、套餐、内容三个视角，用于核对“谁能看什么、为什么能看”。

基础演示数据按真实开课矩阵初始化：小学阶段开设语文、数学、英语，初高中阶段开设 9 门学科，覆盖 12 个年级、2 个学期、2 个阶段，共 288 个学习空间。每个“年级 + 学科 + 学期”提供三种套餐：题、题+讲义、课程+题+讲义。默认演示学生分别覆盖三档套餐，便于直接演示权限差异。

## 教师管理

- 教师管理不是独立账号体系，而是 `users` 中拥有 `teacher` 角色的账号管理视图。
- 教师负责范围继续复用 `teacher_course_scopes` 和 `teacher_class_scopes`；范围变更后，教师下一次请求会按最新仓储数据计算可见学生、课程、作业和资料。
- `campus_admin` 只能管理自己 `campusId` 下的教师；`super_admin` 可以管理全部校区教师。
- 停用教师只更新 `account_status=停用`，不物理删除。鉴权中间件解析 token 后会重新读取用户状态，账号被停用后旧 token 也会立即失效。
- 首次微信绑定采用 `open_id` 为空的待绑定教师档案：登录时未命中 `openId`，可通过手机号匹配教师档案并写入当前微信 `openId`。本地演示仍保留 `code=teacher`、`code=campus`、`code=super` 等固定入口。
